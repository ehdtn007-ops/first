# 매물 관리 규칙

> 이 규칙은 매물 데이터 작업(등록/수정/삭제)에만 적용된다.
> 홈페이지 코드 수정과는 완전히 분리되어 있다. → homepage.md 참조

## 핵심 규칙

1. **묻지 않고 완료까지 한 번에 진행한다.** 중간에 "할까요?", "진행할까요?", "맞나요?" 금지.
2. **"매물번호 N" = `no` 필드가 N인 문서.** `lid`(문서 ID)와 다르다. 반드시 `no` 필드로 검색.
3. **완료 후 결과 브리핑 필수.** 성공/실패, 변경 내용, 홈페이지 매물 링크 포함.

### ⛔ lid ≠ 매물번호 (절대 불변 규칙, 위반 시 작업 전체 무효)

> **이 규칙은 어떤 상황에서도 예외 없이 지켜야 한다. 한 번이라도 위반하면 해당 작업 전체가 무효다.**

- `lid` = Firestore 문서 ID. 내부 식별용. 예: `b7`, `listing_1774195310594`
- `no` = 매물번호. 사용자가 말하는 번호. 예: `7`, `49`
- **이 둘은 완전히 다른 값이다.**
- `lid`가 `b7`이라고 해서 매물번호 7이 **절대 아니다.** `b7`의 매물번호는 `no` 필드에 있는 값이다.
- 매물번호로 요청이 오면 → `no` 필드로 검색. lid로 추측하면 **다른 매물을 건드리게 된다.**
- lid를 수정하거나 형식을 바꾸면 **지도 핀 연동이 깨진다.** 절대 건드리지 않는다.
- lid 형식(`b1`, `listing_xxx` 등)은 있는 그대로 유지. 숫자로 바꾸거나, 접두어를 제거하거나, 어떤 식으로든 변경 금지.

❌ **금지 예시 (이렇게 하면 안 됨)**
- ❌ "매물번호 7의 매매가를 수정하겠습니다. 진행할까요?"
- ❌ "register-listing.js 파일을 읽겠습니다. 진행할까요?"
- ❌ 매물번호 7 요청에 문서 ID `b7`을 수정하는 것 (no 필드 검색 안 하고 lid로 추측)

✅ **올바른 예시 (이렇게 해야 함)**
- ✅ 바로 Firestore에서 `no == 7` 검색 → 해당 문서 수정 → 결과 브리핑

## 결과 브리핑 형식

> **수정 완료**
> - 매물: lid (매물명, 매물번호 N)
> - 변경: 매매가 60,000만원 (6억)
> - 확인: [매물 링크](https://ehdtn007-ops.github.io/first/?lid={lid})

**홈페이지 매물 링크 형식:** `https://ehdtn007-ops.github.io/first/?lid={lid}`

## 매물등록 ("매물등록"이라고 하면)

1. `listing/register-listing.js` 파일을 매물 정보에 맞게 수정한다.
2. `node listing/register-listing.js` 실행하여 Firestore에 직접 등록한다.
3. 등록 완료 후 홈페이지 매물 링크와 결과를 알려준다.
4. 입주가능일은 무조건 "협의".
5. 매물 상세설명은 직접 매물정보를 분석해서 장점을 최대한 간단 명료하게 적는다.(수익률은 적지마)
   - 사용자가 준 정보만으로 설명이 빈약하면(2줄 이하), 매물 정보(위치, 방 수, 층수, 시설, 교통 등)를 바탕으로 자연스러운 약식 문장을 추가한다. 예: "주안역 도보 5분. 대로변 코너 위치, 유동인구 풍부." 같은 식으로.
6. 담당자 이름 적어주면 그 담당자 이름으로 매물 등록. 담당자를 따로 말하지 않으면 기본값은 "마스터".
7. 좌표(lat/lng)는 주소 기반으로 웹 검색하여 찾는다.
8. 매물을 다 등록하면 등록된 매물 링크를 결과와 함께 알려준다.

### 매물등록 필수 루틴 (반드시 모두 수행, 하나라도 빠지면 안 됨)
> `register-listing.js`를 쓰든 인라인 스크립트를 쓰든 아래 항목은 전부 포함해야 한다.

1. **매물번호(`no`) 자동 부여** — Firestore `base_listings` 전체 조회 → 현재 최대 `no` + 1
2. **건축물대장 자동 조회** — `fetchBldgInfo(prov, area, dong, jibun, floor)` 호출하여 totalfloor, parking, mainPurpose, year, approvedate, area2, illegal 자동 입력
3. **좌표(lat/lng)** — 주소 기반 웹 검색
4. **홈페이지 매물 링크** — `https://ehdtn007-ops.github.io/first/?lid={lid}` 형식으로 결과 브리핑에 포함
5. **방향(`direction`)** — 건축물대장에 없음. 사용자가 제공하면 그대로 입력. 제공하지 않으면 `['동향','서향','남향','북향','남동향','남서향','북동향','북서향']` 중 랜덤 선택하여 입력

❌ 인라인 스크립트로 빠르게 등록한다고 건축물대장 조회나 매물번호를 빠뜨리면 안 됨

## Firebase 접속 방법

- **인증 방식:** 익명 인증 (`signInAnonymously`) — 이메일/비번 아님
- **사용 SDK:** `firebase` npm 패키지 (modular v9+)
- **등록 스크립트:** `listing/register-listing.js` (수정 후 `node listing/register-listing.js` 실행)
- **컬렉션:** `base_listings`
- **문서 ID = lid** (자동생성: `listing_` + timestamp). lid는 매물번호가 아니다. 절대 혼동하지 말 것.
- 웹사이트 로그인은 Firebase Auth가 아닌 자체 ID/PW 체크 (마스터: master/master123 하드코딩)

## 매물등록 의미 파악하기

1. 1500/150/30 이런 형식은 순서대로 보증금/월세/관리비.
2. 합 1억8500이면 보증금 + 권리금이 1억8500만이라는 말. 그렇다면 보증금 1500. 권리금 17000만원.
   - "입금가"로 나오면: 권리금 = 입금가 + 1000만원. 등록 시 권리금 필드에 계산된 권리금을 적는다. (예: 입금가 5천 → 권리금 6천)
   - 비밀메모(lmemo)는 사용자가 "비밀메모"라고 명시한 내용만 적는다. 임의로 추가하지 않는다.
2. 보증금과 권리금 관리비 등의 내용이 들어가면 월세. 매매가 얼마 이렇게만 나오면 매매.
3. **매매에는 권리금이 없다.** 매매가만 있다. 권리금은 임대(월세)에만 존재한다.
4. 욕실수는 원룸 갯수대로. 올원룸 20개면 욕실 20개. 혼합 원룸 10개 미니 10개면 욕실 10개. 올미니면 3개 고정.

## 매물 주의사항

- ⛔ **lid/no 혼동 금지** — 위 「lid ≠ 매물번호」 절대 불변 규칙 참조. 여기서 또 반복: lid 건드리면 지도 핀 깨진다. 절대 수정 금지.
- `lat` / `lng` 직접 수정 좌표 정확도 예시 '37.55615569989155' / '126.9264564311379'
- `hidden: true` 로 바꾸면 지도에서 즉시 숨겨짐

## 매물 문제 해결

| 증상 | 원인 | 해결 |
|---|---|---|
| 지도에 핀이 안 뜸 | lat/lng 없음 | 관리자 페이지에서 주소 재저장 |
| 매물이 목록에 안 보임 | hidden: true | Firestore에서 hidden 필드 false로 변경 |
| 가격이 이상하게 표시 | 단위 오류 (원 vs 만원) | price 필드는 만원 단위로 입력 |
| 등록 후 지도에 바로 안 나타남 | 좌표 계산 딜레이 | 30초 후 새로고침 |
